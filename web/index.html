How these are calculated
  • Volume = Σ(weight_lbs × reps) across every set.
  • Volume (tons) = volume_lbs / 2000.
  • Time in gym = end_time − start_time, summed per session.
  • Longest streak = longest run of consecutive calendar days with ≥ 1 logged session.
  • Top exercise = exercise with the most logged sets in the selected year.

    How heat levels are bucketed

    For the selected year, each day's volume is divided by that year's highest-volume day, then bucketed into quartiles:

    • L1 < 25%, L2 < 50%, L3 < 75%, L4 ≥ 75% of the year's peak.
    • Rest days render as L0.

      Apple Health data flow
      • iPhone Health app's "Export All Health Data" produces export.xml (often 500MB+). Drop it at apple_health_export/export.xml.
      • Parsed into three SQLite tables: health_workouts, health_metric_samples (weight / RHR / HRV / BMI), and health_daily (per-day steps, kcal, walking distance, flights).
      • Lifting sessions that Hevy double-writes to Apple Health are filtered out of the cardio view (source NOT IN ('Hevy', 'Strong', …)) so they don't count twice.
      • Exposed to the chat as cardio_summary, body_metric_trend, and daily_activity.

      How this reaches the model

      Non-empty fields get appended to the chat agent's system prompt on every turn, so it can tailor advice without making a tool call. Stored in SQLite (user_profile, single row) — see app/profile.py.

      No routine saved yet. Ask the chat for a split — e.g. "build me a PPL hypertrophy routine".

      Routine vs Planned
      • Routine = a reusable weekly template (Push A / Pull A / Legs A…) with no dates. One slot, stored in the routine table.
      • Planned = specific dated sessions, built from the routine when the agent calls schedule_workout for each day.
      • The routine name + session titles are injected into the system prompt every turn, so follow-ups like "now schedule next week from it" stay consistent.
