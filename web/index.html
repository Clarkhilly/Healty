How these are calculated
  • Volume = Σ(weight_lbs × reps) across every set.
  • Volume (tons) = volume_lbs / 2000.
  • Time in gym = end_time − start_time, summed per session.
  • Longest streak = longest run of consecutive calendar days with ≥ 1 logged session.
  • Top exercise = exercise with the most logged sets in the selected year.

    How heat levels are bucketed

    For the selected year, each day's volume is divided by that year's highest-volume day, then bucketed into quartiles:

    • L1 < 25%, L2 < 50%, L3 < 75%, L4 ≥ 75% of the year's peak.
    • Rest days render as L0.

      Apple Health data flow
      • iPhone Health app's "Export All Health Data" produces export.xml (often 500MB+). Drop it at apple_health_export/export.xml.
      • app/apple_health.py streams it with xml.etree.ElementTree.iterparse (constant memory) and writes three SQLite tables: health_workouts, health_metric_samples (weight / RHR / HRV / BMI), and health_daily (per-day sums of steps, kcal, walking distance, flights).
      • Strength sessions Hevy double-writes into Apple Health are imported but filtered from the cardio view via source NOT IN ('Hevy', 'Strong', …) so they don't count twice against your lifting tables.
      • The chat tools cardio_summary, body_metric_trend, and daily_activity expose this data to the LLM.

      Planned agent

      No upcoming sessions yet. Ask the chat to build a program.

      How planning works
      • The chat tool schedule_workout(date, title, exercises) is the app's first write tool. The LLM calls it once per session when you ask for a program. UPSERTs on (date, title), so re-planning the same day is idempotent.
      • Stored in the planned_workouts table (date, title, JSON exercise list, status). Exposed via GET /api/planned?days_ahead=14 and DELETE /api/planned.
      • Read-only sibling tools: list_planned_workouts lets the model check existing plans before adding new ones; clear_planned_workouts lets it wipe a window before re-planning.

      Workout routine agent

      No routine saved yet. Ask the chat for a split — e.g. "build me a PPL hypertrophy routine".

      Routine vs Planned
      • Routine = reusable weekly template (Push A / Pull A / Legs A…) with no calendar dates. One slot per user, stored in the routine table (single row, id = 1).
      • Planned = specific dated sessions for the upcoming week(s). Built from the routine by calling schedule_workout per day.
      • Chat tools: save_routine(name, sessions) overwrites this slot; get_routine() reads it (the model uses this before applying the routine to the calendar); clear_routine() wipes it. Surfaced via GET /api/routine and DELETE /api/routine.
      • The active routine name + session titles are injected into the system prompt every turn, so the model can apply it consistently on follow-ups like "now schedule next week".
